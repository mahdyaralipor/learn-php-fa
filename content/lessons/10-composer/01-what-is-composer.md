# Composer چیست؟

Composer مدیر وابستگی استاندارد PHP است. یعنی به پروژه می‌گویی چه کتابخانه‌هایی لازم داری، Composer آن‌ها را با نسخهٔ مناسب نصب می‌کند و autoload لازم را هم برایت می‌سازد.

اگر بخواهیم خیلی خودمانی بگوییم: Composer همان چیزی است که جلوی پروژهٔ بی‌نظم، پوشهٔ `lib/` عجیب، دانلود دستی فایل‌ها و جنگ نسخه‌ها را می‌گیرد.

## قبل از Composer چه مشکلی داشتیم؟

فرض کن می‌خواهی از یک کتابخانه برای logging استفاده کنی. در حالت سنتی ممکن بود این کارها را بکنی:

- بروی GitHub و zip دانلود کنی
- فایل‌ها را دستی داخل پروژه کپی کنی
- برای هر کلاس `require_once` بنویسی
- بعداً یادت برود نسخهٔ دقیق چه بوده
- روی سیستم هم‌تیمی یا سرور، نسخهٔ دیگری از همان کتابخانه قرار بگیرد

این روش برای پروژهٔ جدی، خیلی زود دردسرساز می‌شود.

## Composer چه چیزی را حل می‌کند؟

Composer این مشکلات را با چند ایدهٔ ساده حل می‌کند:

- وابستگی‌ها در یک فایل به نام `composer.json` تعریف می‌شوند
- نسخه‌ها طبق constraint مشخص می‌شوند
- پکیج‌ها از مخزن استاندارد نصب می‌شوند
- autoload استاندارد تولید می‌شود
- وضعیت نسخه‌های نصب‌شده در `composer.lock` ثبت می‌شود

## Packagist چیست؟

`Packagist` مخزن اصلی پکیج‌های Composer است.

آدرسش این است:

- [https://packagist.org/](https://packagist.org/)

وقتی در `composer.json` می‌نویسی مثلاً `monolog/monolog`، Composer معمولاً آن را از Packagist پیدا می‌کند.

## یک مثال واقعی

مثلاً اگر بخواهی `vlucas/phpdotenv` را برای خواندن متغیرهای محیطی نصب کنی:

```bash
composer require vlucas/phpdotenv
```

یا اگر بخواهی PHPUnit را فقط برای توسعه و تست نصب کنی:

```bash
composer require --dev phpunit/phpunit
```

با همین دو دستور، هم پکیج نصب می‌شود، هم `composer.json` به‌روزرسانی می‌شود، هم autoload پروژه آماده می‌شود.

## چرا Composer فقط برای پروژه‌های بزرگ نیست؟

بعضی‌ها فکر می‌کنند Composer فقط وقتی لازم است که پروژه خیلی بزرگ شود. این نگاه معمولاً اشتباه است.

حتی در یک پروژهٔ کوچک هم Composer این مزیت‌ها را می‌دهد:

- autoload تمیز
- مدیریت نسخهٔ پکیج‌ها
- اجرای سادهٔ ابزارهای dev
- ساختار قابل‌فهم برای خودت و دیگران

یعنی حتی اگر فعلاً فقط یک پکیج dev نصب کنی، باز هم Composer ارزشش را دارد.

## Composer جای PHP را نمی‌گیرد

یک سوءتفاهم رایج این است که وقتی Composer وارد شد، انگار خود زبان مهم نیست. نه. Composer فقط ابزار مدیریت پروژه است. هنوز هم باید خودت PHP را خوب بلد باشی، namespace را بفهمی، فایل‌ها را درست بچینی و API کتابخانه‌ها را درست استفاده کنی.

Composer نظم می‌دهد، اما فکر کردن را از تو نمی‌گیرد.

## بعد از نصب پکیج چه می‌شود؟

معمولاً این اتفاق‌ها می‌افتد:

- فایل `composer.json` ساخته یا به‌روزرسانی می‌شود
- پوشهٔ `vendor/` ایجاد می‌شود
- فایل `vendor/autoload.php` ساخته می‌شود
- فایل `composer.lock` ثبت می‌کند چه نسخه‌هایی واقعاً نصب شده‌اند

در بیشتر پروژه‌ها، نقطهٔ ورود autoload این فایل است:

```php
<?php

require_once __DIR__ . '/../vendor/autoload.php';
```

## چیزهایی که باید از همین اول عادت کنی

- پکیج را دستی داخل پروژه کپی نکن
- پوشهٔ `vendor/` را خودت ویرایش نکن
- اول API رسمی پکیج را بخوان، بعد استفاده کن
- برای نصب ابزارهای توسعه از `--dev` استفاده کن

## تمرین

1. به Packagist سر بزن و دو پکیج معروف PHP پیدا کن.
2. فکر کن در یک پروژهٔ شخصی کوچک، چه چیزهایی را احتمالاً از Composer نصب می‌کنی.
3. با زبان خودت توضیح بده چرا دانلود zip و کپی‌کردن فایل‌ها از Composer ضعیف‌تر است.

## جمع‌بندی

Composer ستون فقرات پروژهٔ مدرن PHP است. با کمک آن وابستگی‌ها قابل‌مدیریت می‌شوند، autoload استاندارد می‌گیری و پروژه‌ات روی سیستم‌های مختلف قابل‌تکرار می‌شود. در درس بعدی `composer.json` و دستورهای اصلی کار را می‌بینیم.
