# محدودیت نسخه‌ها و `composer.lock`

بخش مهم مدیریت وابستگی فقط «نصب پکیج» نیست؛ باید بدانی چه نسخه‌ای از هر پکیج را می‌خواهی و چطور جلوی به‌هم‌ریختگی نسخه‌ها را بگیری.

در Composer این موضوع با دو چیز کنترل می‌شود:

- version constraints در `composer.json`
- فایل `composer.lock`

## constraint یعنی چه؟

مثلاً وقتی می‌نویسی:

```json
{
  "require": {
    "monolog/monolog": "^3.0"
  }
}
```

داری به Composer می‌گویی هر نسخه‌ای از سری ۳ که با این constraint سازگار است مجاز است.

## نسخهٔ exact

```json
"monolog/monolog": "3.5.0"
```

یعنی دقیقاً همین نسخه.

### مزیت

- رفتار کاملاً قابل‌پیش‌بینی

### عیب

- خیلی خشک است
- patchهای مفید را خودکار نمی‌گیری

## عملگر `^`

```json
"monolog/monolog": "^3.0"
```

این رایج‌ترین انتخاب در پروژه‌های مدرن است.

معمولاً یعنی:

- از `3.0.0` به بالا
- تا قبل از `4.0.0`

یعنی patch و minorهای سازگار را می‌پذیرد، ولی major بعدی را نه.

## عملگر `~`

```json
"guzzlehttp/guzzle": "~7.8"
```

معمولاً یعنی:

- از `7.8.0` به بالا
- تا قبل از `8.0.0`

و اگر روی بخش دقیق‌تری بنویسی مثلاً `~7.8.2`، بازه محدودتر می‌شود.

## کدام بهتر است: `^` یا `~`؟

در بیشتر پروژه‌ها `^` انتخاب طبیعی‌تری است، چون اجازه می‌دهد آپدیت‌های سازگار را بگیری. `~` کمی محافظه‌کارتر است و در بعضی سناریوها ترجیح داده می‌شود. مهم‌تر از حفظ‌کردن جزئیات، فهمیدن ایده است:

- major جدید می‌تواند breaking change بیاورد
- minor و patch معمولاً امن‌ترند، ولی باز هم باید تست داشته باشی

## چرا نباید بی‌فکر `*` یا بازه‌های شل بگذاری؟

مثلاً این انتخاب خوبی نیست:

```json
"some/package": "*"
```

چون عملاً داری می‌گویی هر نسخه‌ای قبول است. این در پروژهٔ واقعی یعنی «سورپرایزهای آینده را خوش‌آمد می‌گویم».

## `composer.lock` چه کار می‌کند؟

`composer.json` می‌گوید چه نسخه‌هایی مجازند.

`composer.lock` می‌گوید چه نسخه‌هایی واقعاً نصب شده‌اند.

این تفاوت خیلی مهم است.

مثلاً ممکن است در `composer.json` بنویسی:

```json
"phpunit/phpunit": "^11.0"
```

اما داخل `composer.lock` ثبت شده باشد که الان نسخهٔ دقیق `11.5.3` نصب شده است.

## چرا lock file مهم است؟

اگر `composer.lock` را commit کنی:

- سیستم تو و هم‌تیمی‌ات یک نسخهٔ دقیق می‌گیرند
- محیط CI همان نسخه را می‌گیرد
- سرور deploy هم همان نسخه‌ها را می‌گیرد
- باگ‌های «روی سیستم من کار می‌کرد» کمتر می‌شوند

## آیا همیشه باید `composer.lock` را commit کنیم؟

برای اپلیکیشن‌ها: معمولاً بله.

برای کتابخانه‌ها: موضوع ظریف‌تر است، چون کتابخانه می‌خواهد با بازه‌ای از نسخه‌ها سازگار بماند. اما در این دوره تمرکز ما روی اپلیکیشن است، پس قاعدهٔ عملی ما ساده است:

- `composer.json` را commit کن
- `composer.lock` را هم commit کن

## workflow سالم برای آپدیت وابستگی‌ها

1. constraint مناسب بنویس، معمولاً با `^`
2. عمداً و آگاهانه `composer update` بزن
3. تست‌ها را اجرا کن
4. تغییرات `composer.lock` را commit کن

## نشانهٔ خطر

اگر بدون دلیل هر روز `composer update` می‌زنی و نمی‌دانی چه چیزی تغییر کرده، عملاً داری ریسک تولید می‌کنی.

## مثال ذهنی

فرض کن constraint تو این است:

```json
"vlucas/phpdotenv": "^5.6"
```

امروز نسخهٔ `5.6.2` نصب می‌شود. دو ماه بعد `5.6.5` یا `5.7.0` هم ممکن است مجاز باشد. اگر فقط `composer install` بزنی و lock file را نگه داری، همان نسخهٔ ثبت‌شده نصب می‌شود. اگر `composer update` بزنی، ممکن است نسخهٔ جدیدتری داخل همان بازه بگیری.

## تمرین

1. تفاوت نقش `composer.json` و `composer.lock` را با یک مثال توضیح بده.
2. بگو چرا `^8.2` برای نسخهٔ PHP رایج‌تر از نسخهٔ exact است.
3. فرض کن پروژه روی CI شکست خورده چون هم‌تیمی‌ات `composer.lock` را commit نکرده؛ توضیح بده چه اتفاقی افتاده است.
4. دو constraint فرضی با `^` و `~` بنویس و تفاوت کلی آن‌ها را توضیح بده.

## جمع‌بندی

version constraintها مشخص می‌کنند چه نسخه‌هایی مجازند و `composer.lock` مشخص می‌کند دقیقاً چه نسخه‌هایی نصب شده‌اند. برای اپلیکیشن‌های PHP، فهم این دو مورد یکی از مهم‌ترین عادت‌های حرفه‌ای است؛ چون مستقیماً روی پایداری پروژه، reproducibility و امنیت deploy اثر می‌گذارد.
