# چرا تست می‌نویسیم؟

اولین پاسخ ساده این است: برای اطمینان. اما اطمینان دقیقاً از چه چیزی؟

- اینکه کد امروز درست کار می‌کند
- اینکه تغییر فردا رفتار دیروز را نشکند
- اینکه باگ گزارش‌شده دوباره تکرار نشود
- اینکه با خیال راحت‌تر refactor کنی

## تست، بیشتر از پیدا کردن باگ است

خیلی‌ها فکر می‌کنند تست فقط برای شکار باگ است. تست این کار را هم می‌کند، اما ارزش اصلی‌اش در «اعتماد به تغییر» است.

در پروژهٔ واقعی، کدی که نمی‌توانی با خیال راحت تغییرش بدهی، کم‌کم تبدیل می‌شود به بخش ترسناک سیستم.

## تست واحد و تست یکپارچه

### تست واحد (Unit Test)

روی یک واحد کوچک از منطق تمرکز می‌کند. معمولاً:

- یک کلاس
- یک متد
- یک تابع

مثلاً تست متد `add()` در یک `Calculator`.

### تست یکپارچه (Integration Test)

بررسی می‌کند چند بخش کنار هم درست کار می‌کنند.

مثلاً:

- repository واقعاً به دیتابیس وصل می‌شود
- controller با view و request درست هماهنگ است
- API واقعاً JSON درست برمی‌گرداند

## فرق ذهنی این دو

تست واحد می‌پرسد:

- «این منطق کوچک، به‌تنهایی درست است؟»

تست یکپارچه می‌پرسد:

- «وقتی چند بخش واقعی را کنار هم می‌گذارم، هنوز همه‌چیز درست است؟»

هر دو مهم‌اند. اگر فقط unit test داشته باشی، ممکن است اتصال واقعی بخش‌ها را از دست بدهی. اگر فقط integration test داشته باشی، پیدا کردن منشأ خطا سخت‌تر و اجرای تست‌ها کندتر می‌شود.

## چه چیزهایی برای شروع ارزش تست دارند؟

برای پروژه‌های آموزشی و کوچک، معمولاً این‌ها بهترین شروع هستند:

- منطق محاسباتی
- validation rules
- formatterها و parserها
- serviceهایی با شرط و شاخهٔ منطقی واضح
- رفتارهای امنیتی مهم

## چه چیزهایی معمولاً تست کم‌ارزش تولید می‌کنند؟

- getter و setter کاملاً بدیهی
- تستی که فقط implementation فعلی را کپی کرده
- تستی که برای هر تغییر ظاهری می‌شکند
- تستی که از کد واقعی کمتر خواناست

## تست و اعتماد تیمی

وقتی تست خوب داری، هم‌تیمی‌ات لازم نیست برای هر تغییر کوچک همه‌چیز را دستی چک کند. CI تست‌ها را اجرا می‌کند و اگر چیزی شکسته باشد، زود متوجه می‌شوی.

## تست، طراحی را هم بهتر می‌کند

کدی که سخت تست می‌شود، خیلی وقت‌ها بوی طراحی نامناسب می‌دهد. مثلاً:

- وابستگی‌ها زیاد و درهم‌اند
- کلاس همه‌کاره است
- side effectها همه‌جا پخش شده‌اند

تست‌پذیری معمولاً ما را هل می‌دهد به سمت طراحی تمیزتر.

## یک مثال ذهنی

فرض کن متدی داری که قیمت نهایی سفارش را با تخفیف و مالیات حساب می‌کند. اگر این منطق را تست نکنی، هر تغییر کوچک ممکن است بدون اینکه بفهمی عدد اشتباه تولید کند. اما با چند تست دقیق، می‌فهمی در سناریوهای اصلی هنوز درست کار می‌کند.

## تمرین

1. سه بخش از یک پروژهٔ PHP فرضی را نام ببر که برای شروع ارزش unit test دارند.
2. یک مثال از integration test در پروژهٔ وب بنویس.
3. با زبان خودت توضیح بده چرا تست فقط برای پیدا کردن باگ نیست.

## جمع‌بندی

تست یعنی ساختن شبکهٔ اطمینان. unit test روی بخش کوچک تمرکز می‌کند، integration test روی همکاری بخش‌ها. هر دو کنار هم اعتماد می‌سازند و توسعه را سریع‌تر و کم‌استرس‌تر می‌کنند.
