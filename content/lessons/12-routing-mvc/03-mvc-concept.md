# MVC به زبان ساده

`MVC` مخفف `Model-View-Controller` است. هدفش این نیست که پروژه را «شیک» کند؛ هدفش این است که مسئولیت‌ها را از هم جدا کند تا کد قابل‌فهم و قابل‌نگهداری بماند.

## سه بخش اصلی

### Model

مسئول داده و منطق مرتبط با آن است.

مثلاً:

- `PostRepository`
- `UserRepository`
- `TaskService`

Model لازم نیست حتماً active record باشد. در این دوره، هر لایه‌ای که با داده و منطق دامنه کار کند، در این خانواده قرار می‌گیرد.

### View

مسئول نمایش است.

مثلاً:

- قالب HTML
- نمایش عنوان پست
- نمایش خطاها یا پیام موفقیت

View نباید جای منطق سنگین کسب‌وکار باشد.

### Controller

بین درخواست و پاسخ قرار می‌گیرد.

کارهای معمول controller:

- دریافت request
- صدا زدن service یا repository
- آماده‌کردن داده برای view
- انتخاب response مناسب

## مثال ذهنی

در مسیر `/posts`:

- controller تصمیم می‌گیرد چه کار شود
- model دادهٔ پست‌ها را می‌آورد
- view آن‌ها را رندر می‌کند

## چرا جداسازی مهم است؟

اگر همه‌چیز را در یک فایل بریزی، بعد از مدتی این‌ها قاطی می‌شوند:

- HTML
- query دیتابیس
- validation
- redirect
- session
- شرط‌های امنیتی

نتیجه‌اش فایلی است که هم تغییرش سخت است، هم تستش سخت است، هم پیدا کردن باگ در آن خسته‌کننده است.

## نمونهٔ بد

```php
<?php

$pdo = new PDO(...);
$stmt = $pdo->query('SELECT * FROM posts');
$posts = $stmt->fetchAll();

foreach ($posts as $post) {
    echo '<h2>' . htmlspecialchars($post['title']) . '</h2>';
}
```

این کد شاید کار کند، اما مسئولیت‌ها را مخلوط کرده است.

## نسخهٔ بهتر از نظر تفکیک

- `PostRepository` داده را می‌آورد
- `PostController` تصمیم می‌گیرد چه چیزی نمایش داده شود
- `views/posts/index.php` HTML را رندر می‌کند

## MVC یعنی قانون خشک؟

نه. MVC بیشتر یک راهنماست تا یک دین رسمی. پروژه‌های کوچک می‌توانند نسخهٔ ساده‌تری از آن داشته باشند. مهم‌تر از اسم‌ها، جداسازی مسئولیت‌هاست.

## Controller نباید همه‌کاره شود

یک خطر رایج این است که controller تبدیل شود به فایل همه‌چیزدان:

- validation
- query مستقیم
- ساخت HTML
- auth
- logging

این یعنی MVC را فقط در اسم اجرا کرده‌ای. controller باید هماهنگ‌کننده باشد، نه انبار منطق پروژه.

## تمرین

1. برای یک صفحهٔ «نمایش لیست پست‌ها» بگو هر بخش MVC چه مسئولیتی دارد.
2. سه نشانه بنویس که بفهمی یک controller بیش از حد چاق شده است.
3. یک مثال از کدی بزن که بهتر است داخل view نباشد.

## جمع‌بندی

MVC یعنی جداکردن مسئولیت داده، نمایش و هماهنگی درخواست. این جداسازی پروژه را قابل‌فهم‌تر و قابل‌گسترش‌تر می‌کند. در درس بعدی یک controller و view واقعیِ ساده را کنار هم می‌گذاریم.
