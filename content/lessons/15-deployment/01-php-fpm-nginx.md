# Nginx و PHP-FPM به زبان ساده

در محیط production معمولاً PHP مستقیماً مثل یک برنامهٔ مستقل برای هر request اجرا نمی‌شود. یکی از الگوهای بسیار رایج این است:

- Nginx درخواست را می‌گیرد
- فایل‌های static را خودش سرو می‌کند
- requestهای PHP را به `PHP-FPM` می‌فرستد

## نقش Nginx

Nginx معمولاً مسئول این کارهاست:

- دریافت اتصال HTTP
- سرو کردن فایل‌های static مثل CSS و تصویر
- rewrite یا route اولیه
- فرستادن فایل‌های PHP به PHP-FPM

## نقش PHP-FPM

`PHP-FPM` یعنی FastCGI Process Manager.

به زبان ساده:

- چند worker آماده نگه می‌دارد
- کد PHP را برای requestها اجرا می‌کند
- نتیجه را به وب‌سرور برمی‌گرداند

## جریان یک request

برای مثال وقتی کاربر `/posts` را باز می‌کند:

1. مرورگر درخواست را به سرور می‌فرستد
2. Nginx درخواست را می‌گیرد
3. اگر فایل static نباشد، آن را به `index.php` و PHP-FPM هدایت می‌کند
4. PHP-FPM کد برنامه را اجرا می‌کند
5. خروجی به Nginx برمی‌گردد
6. Nginx پاسخ را به کاربر می‌دهد

## چرا این مدل خوب است؟

- Nginx در سرو کردن فایل‌های static قوی است
- PHP-FPM برای اجرای اسکریپت‌های PHP مدیریت‌شده‌تر عمل می‌کند
- مقیاس‌پذیری و کنترل بهتری می‌گیری

## مفهوم `public/`

در پروژه‌های PHP بهتر است document root روی پوشهٔ `public/` تنظیم شود، نه ریشهٔ کل پروژه. این کار باعث می‌شود فایل‌های داخلی مثل:

- `composer.json`
- `src/`
- `.env`

مستقیماً از وب قابل‌دسترسی نباشند.

## ایدهٔ کلی پیکربندی

لازم نیست فعلاً همهٔ جزئیات Nginx را حفظ باشی. فقط این تصویر ذهنی مهم است:

- root روی `public/`
- اگر فایل واقعی نبود، درخواست به `index.php`
- فایل‌های `.php` از طریق FastCGI به PHP-FPM

## تمرین

1. با زبان خودت فرق نقش Nginx و PHP-FPM را توضیح بده.
2. بگو چرا `public/` باید document root باشد.
3. مسیر عبور یک request را از مرورگر تا اجرای PHP مرحله‌به‌مرحله شرح بده.

## جمع‌بندی

در deployهای رایج PHP، Nginx در ورودی می‌ایستد و PHP-FPM اجرای کد را انجام می‌دهد. فهم این همکاری کمک می‌کند پیکربندی production برایت کمتر جادویی به نظر برسد.
